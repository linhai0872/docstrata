层：prd（产品主张，前瞻 intent）— 对内的产品定位/价值/功能范围/roadmap。非 CoALA 记忆层，记的是当下主张而非已沉淀事实。

## 维度校准（PRD 专属，覆盖 _judge.md 的通用解释）

PRD 是主张/intent，不是可追溯到代码的事实。据此调整两个通用维度的判读，不要按其余层的标准误判：

- **来源忠实度（source_fidelity）**：对 PRD，"来源"是用户真实表达的定位/意图，**不是代码**。陈述未来 roadmap、尚未实现的功能**不算编造**——intent 本就指向还没做的事。只有当 PRD 把"已实现/已约定"的现状说错、或与用户明确表达的定位相矛盾时，才算失真。**严禁因"这个功能在代码里找不到"扣分。**
- **认知状态准确（epistemic_accuracy）**：PRD 全篇是主张，`[事实]/[推断]` 标注不是重点。关键看两件事：是否混入了"伪装成已定事实的愿望"（把未定的写成已定）；是否把历史（已发生的取舍/约定）误当主张写入——那属于 requirements 层，不该在 PRD。

## 契约清单（用于「契约覆盖度」打分）

1. 定位 — 是什么产品、解决谁的什么核心问题、与替代方案的根本差异
2. 目标用户 — 为谁做 + 核心诉求
3. 价值主张与优先级原则 — 优先什么、牺牲什么（要能用于裁决，而非口号）
4. 功能范围 — 主要功能（意图层：做什么 + 为什么值得做，不含验收标准）
5. 非目标 — 明确不做什么 / 不为谁做
6. roadmap — high-level 推进顺序，条目带日期

## 目标受众（用于「受众适配」打分）

对内：产品负责人 + 开发 + coding agent。grilling 卡在产品决策时读它消歧。不是对外营销文案——堆营销话术反而是错位。

## 专属维度（layer_specific，0-3）

**可裁决性 + 事实/主张不互窜**：定位与优先级原则是否写成了能实际指导决策的主张，且没把历史事实与愿望混为一谈。
- 3：定位/优先级原则声明式、可裁决（能回答"该选 X 还是 Y"）；主张与已发生事实清晰分离
- 2：多数可用于决策，个别流于口号；或偶有历史/主张轻微混写
- 1：大量空泛口号难以指导决策；或事实/主张明显混淆
- 0：纯愿景口号、无裁决价值，或通篇把愿望当已定事实陈述
