# Source Criticism — 信息批判方法论

横切方法论：从项目多源信息提炼文档时，如何系统化发现并处理信息本身的质量问题（来源可信度不一、互相矛盾、过期误导、缺失、事实与推断混淆）。

EXPLORE / MAP / GENERATE 三步都引用本文件。理论来源经实测精简，砍掉了过度设计（完整 ACH 八步、Admiralty 双轴打分、Wang&Strong 全维度、W3C PROV 本体），只保留对 agent 有操作价值的核心。

## 第一性原理

**离"机器执行的事实"越近越可信。代码是一手事实，文档是对事实的解读，解读可能错、可能过期。**

## 准则1：来源可信度排序

多个来源描述同一事实时，按可信度取信（高 → 低）：

1. **运行中的代码** — 机器执行即真相
2. **测试**（单元/集成）— 可验证的行为规约
3. **git 历史** — 时间戳化的一手记录（粒度粗，但时效强）
4. **代码注释 / docstring** — 作者意图，可能过期
5. **官方文档**（README / API 文档）— 规约，有漂移风险
6. **设计文档 / ADR** — 决策记录，查时间戳
7. **开发记忆 / 日志** — 当时理解的快照（`.workbuddy/`、`.claude/memory/`、`DEVLOG.md` 等任意形式，按特征归此档，不靠固定路径识别）
8. **口述 / 非正式描述** — 二手解读，必须交叉验证

**例外（按交付物形态调整）**：排序默认假设核心交付物是代码。当核心交付物不是代码时，把该形态视为第 1 档一手事实：
- Dify DSL 应用 → YAML/JSON 配置（workflow/chatflow/agent 定义）是一手事实
- 纯文档项目 → 文档本身是一手事实
- Skill → SKILL.md + 脚本是一手事实
- 判断锚点：**哪个产物被机器真正执行/消费，它就是一手事实。**

## 准则2：矛盾处理

发现冲突时，**绝不静默选一个**。静默选择等于生产误导性文档。

1. 识别冲突类型：
   - **时间冲突**：旧文档 vs 新代码 → 以高可信来源（通常代码）为准，低可信来源标 `[过期: 与代码冲突]`
   - **真矛盾**：不同来源描述不同行为，无法自动判定 → 显式呈现两种说法 + 各自来源，标 `[冲突: A说X / B说Y，需人工确认]`，或进 GRILL 问人
   - **粒度冲突**：概述对、细节错 → 回到代码层验证细节
2. 判断用**证伪思维**：问"哪条证据若为真，会让另一条为假"，诊断价值高的证据优先；不要数"多数支持哪个"。
3. **独立性要求**：同一人/同一时期写的多份文档一致，不算验证：它们可能集体错。只有相互独立的来源一致才提升置信度。

## 准则3：事实 vs 推断标注

**推断不是问题，把推断当事实才是问题。** 看起来权威实则编造的文档，比没有文档更危险。

| 标注 | 含义 | 适用 |
|---|---|---|
| `[事实]` | 可从来源直接读取 | 函数签名、配置值、测试断言、git 记录 |
| `[推断]` | 从事实逻辑推导，有明确前提 | 职责边界、"这段代码目的是…"、因果解释 |
| `[待确认]` | 未经验证的假设 | 无来源支撑、靠常理填的内容 |
| `[过期]` | 有时间戳且与代码矛盾 | 旧设计文档、被否定方案的残留描述 |
| `[缺口: 缺什么 \| 建议来源]` | 信息缺失 | 无测试无文档的行为、找不到背景的决策 |

不要用"通常来说""一般情况下"掩盖信息缺口：缺就标 `[缺口]`。

判断"可直接读取"还是"需要推导"：含"应该/可能/似乎/负责/目的是"的陈述基本都是推断。

## 准则4：打破 LLM 先验偏向（agent 特有）

实测发现：**LLM 倾向用训练数据静默覆盖项目特定信息**，即使上下文明确相反。项目用了非主流实现 / 违反"最佳实践"的刻意设计时，这个偏向会把真相改写成"标准做法"。

处理：
- **上下文信息优先于先验知识**：项目怎么做的就怎么记。
- 代码行为与"常规做法"不符时，**记录实际行为，不要"纠正"它**。
- 疑似异常标 `[注意: 偏离常见模式，验证意图]`，而非改写成标准描述。

## 在五步循环中的应用

- **EXPLORE**：按准则1 给收集到的来源定可信度基线。
- **MAP**：给维度打置信度时，用准则1 的排序；发现跨来源不一致时启动准则2。
- **GENERATE**：按准则3 标注每条信息的认知状态；遇非标实现用准则4。
