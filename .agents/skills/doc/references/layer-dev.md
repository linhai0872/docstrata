# Layer: dev (Procedural)

开发过程中为实现需求而构建的**推断/实践事实**：工程师工作得出的结论，区别于原始事实。包括为达成需求建立的关系、延伸知识、开发测试结论。

最下游层：基于 requirements（要做什么）、knowledge（业务背景）、代码（怎么实现的）综合推导。

产出：`docs/dev.md`

## Completeness Contract（信息维度）

1. **需求→实现的推导链** — 原始需求是如何落成具体系统功能的？中间做了哪些关系构建、延伸推断？
2. **实践得出的事实** — 开发/测试过程中验证出来的结论（性能数据、踩坑、依赖行为、边界条件）。
3. **已验证/已否定的方案** — 试过什么、成了什么、放弃了什么。防止后人重蹈覆辙。

## 探索重点

- 上游：读 requirements（需求意图）+ knowledge（业务规则）作为推导起点。
- 代码：实现细节、关键算法、数据流、外部依赖的实际行为。
- 历史：git log、测试文件、CHANGELOG 里"为什么改"的线索。
- **明确区分事实层级**：原始事实标明出处；推断/实践结论标注 `[推断]` 或 `[实测]`。

**「实践结论」vs「延伸知识」判断标准**：实践结论是本项目验证出的、带具体数字/代码/边界的事实（如"重试 3 次因 API 偶发超时""参数 X=78pt 经校准"）；延伸知识是为完成需求而习得的、相对通用的领域规律或外部系统行为（如"某外部 API 的分页从 0 起""该格式的水印 XML 结构"）。前者绑定本项目，后者换个项目仍成立。

## Section 骨架

```markdown
# Development Notes — {项目名}

> {一句话：这个项目在实现过程中沉淀了哪些工程结论}

last-verified: YYYY-MM-DD

## 需求到实现的映射
{原始需求 → 实际系统功能的推导链。每条：需求点 → 怎么实现的 → 为什么这样实现}

## 关键实现决策
{实现层面的技术选择与理由。比 requirements 的 ADR 更细，聚焦"怎么做到"}

## 实践结论
{开发/测试验证出的事实：性能、边界、依赖行为、踩坑。标注 [实测]/[推断]}

## 已否定的方案
{试过但放弃的实现路径 + 原因。防重蹈覆辙}

## 延伸知识
{为完成需求而构建的额外认知：领域关系、外部系统行为、约束的来源}
```

## Grill 示例（仅缺口）

- 实践结论 low：「代码里对这个外部 API 加了 3 次重试——是踩过坑（它会偶发超时）才加的，还是预防性的？如果有具体故障，值得记下来。」
- 否定方案 missing：「实现这个功能时有没有试过别的方案后放弃的？这类信息对后面维护的人很有价值，但代码里看不出来。」
