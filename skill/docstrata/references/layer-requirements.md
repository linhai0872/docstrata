# Layer: requirements (Episodic)

**需求方与开发者的共识约定 + 开发计划**。记录约定了什么、为何这样决策、谁负责；核心是责任与决策留痕，不只是功能列表。

产出：`docs/requirements.md`

## 与 PRD 的边界（铁律，先读）

requirements 记**事实**，PRD 记**主张**：

- **requirements = 已发生的事实**：需求方真提过什么、双方真约定了什么、真放弃了什么。**只增不改写**——发生过的不因后来变卦而变假。
- **PRD = 当下的主张**：要把产品做成什么、为什么，会随认知更新而重写。
- **铁律：requirements 不写愿望，PRD 不写历史。** 「考虑过 OIDC 但放弃了」是事实 → 留 requirements；「未来要支持多租户」是主张 → 进 PRD。
- **产品级决策上移 PRD**：定位、要不要做某功能、对外/合规策略属 PRD。requirements 只留「这批活的事实共识 + 取舍 + 验收」。PRD 缺位的老项目，这些产品决策常被误塞进下面的决策记录里，生成/更新时识别出来上移到 PRD。

## Completeness Contract（信息维度）

生成前确保以下维度达标（探索 + 必要时 grill）：

1. **原始需求意图** — 需求方到底想解决什么问题？尽量还原原文/原始意图。若项目里有原始需求文档（需求 issue、聊天记录），优先保真引用，不要改写。产品级愿景/定位归 PRD，不在此层。
2. **共识约定** — 需求方与开发者之间确认的边界：做什么、不做什么、以及"怎么算做到了"（验收标准）。这是责任划分的核心。
3. **开发计划** — 开发者基于自己的考虑制定的大致方案/路径（high-level，不是详细设计）。
4. **关键取舍** — 有真实权衡的决策点：选了什么、放弃了什么、为什么。包括被否定的方案。这是防未来 agent 翻案的最高价值信息（README 目标：别让已定决策被当成开放问题重跑）。

## 探索重点

有原始材料就保真还原，没有就从成品倒推并标清楚。

- 找原始材料：`docs/`、issue、PRD、需求相关的原始文档。有则保真还原，无则从成品 + 用户 grill 倒推。
- 区分"原始事实"（需求原文）与"倒推还原"（从代码反推的意图）：后者标注 `[推断]`。

## Section 骨架

```markdown
# Requirements — {项目名}

> {一句话：这个项目是需求方与开发者关于什么的共识}

last-verified: YYYY-MM-DD

## Background
{为什么发起这个项目，需求方的处境与动机}

## Goals
{要达成的目标，可度量优先}

## Non-goals
{明确不做的，划清边界，和 Goals 同等重要}

## Requirements
{需求条目。有原始材料则保真引用；从成品倒推的标注 [推断]}

## Plan
{开发者制定的 high-level 方案/路径}

## 关键决策
{每条四段骨：决策 · 背景 · 放弃的备选 · 后果。只记有真实权衡的。
编号可选（D1、D2…）——若项目已在 commit/对话里用编号当交叉引用 handle，保留编号；纯术语（"ADR"）不必写}

## Constraints
{约束：技术/合规/时间/外部契约等代码里看不出来的}
```

## 决策记录写法（留骨不留黑话）

借 ADR 的**结构纪律**，不引入它的术语。值钱的是四段骨——决策 / 背景 / 放弃的备选 / 后果；「放弃了什么 + 为什么」最防翻案。大白话决策记录往往丢掉「放弃的备选」和「后果」，所以：**保留四段骨，用大白话写，去掉「ADR」缩写仪式。** 编号（D1、D23）若在项目里实际流通（commit/对话引用）就保留当 handle，删了会断引用。

不引入「BDR（Business Decision Record）」——它不是既定实践。产品决策与技术决策的区分用分层解决（产品决策进 PRD，技术/交付决策留此层），比造缩写干净。

## Grill 示例（仅缺口）

- 原始意图 missing：「我没找到原始需求文档。这个项目要解决的核心问题是什么？我从代码推测是 X——对吗？」
- 关键取舍 low：「代码用了 SQLite 而非 Postgres，这是刻意取舍还是默认选择？如果是取舍，放弃 Postgres 的原因是？」
