# Layer: wiki (Semantic)

**系统全景**，让所有人（含业务人员）快速了解并理解这个系统。偏业务的 README，不那么技术导向。回答「这个系统是什么、能做什么」。

产出：`docs/wiki.md`

## 与 PRD 的边界（先读）

wiki 对外、as-built——现在是什么、怎么用；PRD 对内、前瞻 intent——要做成什么。两者在「定位/目标用户/价值」上重叠。**重叠项以 PRD 为源，wiki 引用不复制**（如「定位见 prd.md」或一句话转述 + 链接），否则两处各写一份必然漂移过期。wiki 只在自己独有的事上展开：对外措辞、上手步骤、面向非技术读者的功能解释。若项目无 PRD，wiki 自行覆盖这些维度。

## Completeness Contract（信息维度）

1. **系统存在的理由** — 这个系统为什么存在？解决谁的什么问题？（可引用 requirements 层）
2. **目标用户** — 谁在用？业务角色有哪些？
3. **核心能力边界** — 系统能做什么、不能做什么，主要功能模块。
4. **关键概念** — 理解这个系统必须先懂的领域概念/术语（可引用 knowledge 层 Glossary）。
5. **如何上手** — 业务用户怎么启动/使用这个系统（双击 index.html？跑哪个命令？访问哪个地址？）。读完 wiki 的人应该知道下一步怎么用，这属于 wiki 范畴而非纯 dev 知识。

## 探索重点

上游结论直接引用，从代码提炼能力时用业务语言转译。

- 上游优先：若 `requirements.md` / `knowledge.md` 存在，直接引用其结论，不重复推导。
- 从代码/配置提炼"能力"，但用业务语言转译：业务人员读不懂"REST endpoint"，能读懂"支持批量导出订单"。
- 按项目类型找能力：全栈看路由+页面、Dify 看 workflow 节点做什么、MCP/CLI 看工具/命令清单。

## Section 骨架

```markdown
# {项目名}

> {一句话电梯陈述：这个系统是什么、给谁用、解决什么}

last-verified: YYYY-MM-DD

## 这是什么
{2-3 句话，业务语言，让任何人秒懂}

## 解决什么问题
{背景与价值，面向非技术读者}

## 谁在用
{目标用户/角色}

## 核心功能
{主要能力，业务语言描述，按模块/场景组织。避免技术术语}

## 关键概念
{理解系统必须懂的领域术语。引用 knowledge 层 Glossary 若存在}

## 如何运转（概览）
{高层流程，一图胜千言可用文字流程。不深入技术实现}

## 如何上手
{业务用户的启动/使用方式：双击什么、跑什么命令、访问什么地址}

## 边界
{明确不做什么、不负责什么}
```

## Grill 示例（仅缺口）

- 目标用户 low：「我从代码看到有管理后台和 C 端两套界面——主要面向的用户是内部运营还是外部客户？两者都有的话，哪个是核心？」
- 系统理由 missing：「没有 requirements 文档可参考。用一句话说，这个系统对业务最大的价值是什么？我推测是 X。」
