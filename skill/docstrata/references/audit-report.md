# Audit Report — 信息健康诊断副产物

任何生成动作完成后，向用户输出一份**信息健康诊断报告**——告诉用户"你项目里的事实信息哪些对不上了"。它产出判断，直接在对话里说，默认不落盘。

## 定位

audit 是生成流程的副产物，只产判断，默认不落盘。

- **生成流程的副产物**：跑 wiki/requirements/knowledge/dev/all 时自然带出，不单独成命令。
- **只产判断，默认不落盘**：生成模式已经在写文档，audit 的独特价值是"信任评估"，直接在对话里汇报（用户要求才写文件）。
- 对象是**全部 context**：代码、git 历史、现有文档、规范文件（CLAUDE.md/AGENTS.md/lint 配置等）之间的事实是否自洽。
- **输出时序**：`all` 模式在全部层跑完后统一汇报一份；单层模式在该层生成完即汇报。
- **纯派生层不单独触发 audit**：`index` 不走 EXPLORE/MAP，单跑 `/docstrata index` 时没有新的探索数据，不产 audit（其诊断数据来自其他层的探索）。

## 数据来源

audit 汇总 EXPLORE / MAP 阶段已发现的问题，不重新扫描。来源包括：

- 准则1（可信度排序）筛出的：仅低可信来源支撑、无代码佐证的主张
- 准则2（矛盾处理）标出的：`[过期]` `[冲突]`
- 准则3（事实vs推断）标出的：`[缺口]`
- 探索中遇到的无主孤儿文档（不属于任何层、且与代码无关联的旧文档）

## 报告结构

按严重度排序，每条附证据和建议动作：

```markdown
## 信息健康诊断

扫描范围：{代码 / git / docs/ / 规范文件，列出实际查了什么}

### 冲突（需人工裁决）
- **{主题}**：{来源A} 说 X，{来源B} 说 Y。
  证据：{文件:行 或 具体依据}
  建议：以 {高可信来源} 为准 / 需你确认

### 过期（与当前事实不符）
- **{文件}**：描述了 {旧方案}，但代码已是 {现状}。
  证据：{对比依据}
  建议：更新 / 归档 / 删除

### 版本漂移（同一值在多处不一致）
- **{值名，如版本号/端口/常量}**：{位置A}=X，{位置B}=Y。
  证据：{各位置 文件:行}
  建议：定单一可信源（如 package.json），其余引用它
  （机械漂移，独立于"过期"：过期是内容决策失效，漂移是同一事实多处抄写不同步）

### 缺口（重要但无人记录）
- **{主题}**：{哪一层的哪个维度} 缺信息。
  建议来源：{人 / 文件 / 代码区域}

### 待验证的外部假设（项目内无法证实）
- **{假设}**：{依赖某外部系统/服务的行为，但项目内无证据可验}。
  证据：{该假设的出处}
  建议：实测 / 向了解的人确认

### 无主孤儿文档
- **{文件}**：不属于四层任何一类，且与当前代码无对应。
  建议：归入 {某层} / 归档 / 确认是否废弃
  （大项目无需全量回溯：按文档年龄 + 对应代码改动频率启发式筛查即可，并在"有限度声明"里说明抽样范围）
```

## 纪律

audit 只报告有证据的问题，只诊断不自动修复。

- **不夸大**：没发现问题就如实说"未发现信息质量问题"，不为凑内容编造。
- **给证据**：每条问题必须可追溯到具体来源，不空泛断言。
- **不自动修**：audit 只诊断和建议，不擅自改/删用户的文档（删改是 Ask First 级操作）。
- **有限度声明**：若因范围/权限只扫了部分 context（如没读 .doc、没查 git blame），明说哪些没覆盖，不假装查全了。
