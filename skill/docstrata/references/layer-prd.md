# Layer: prd（产品主张）

**对内产品主张**：要做成什么、为谁做、为何这样定、接下来怎么走。最上游层，requirements 与 dev 的意图源头。

产出：`docs/prd.md`

## 定位（先读，划清三条边界）

PRD 记**前瞻 intent**——要做成什么。其余三层各管一类不同性质的内容：

- **PRD vs requirements**：PRD 记**主张**，认知更新时可重写；requirements 记**事实**——需求方真提过、双方真约定、真放弃过的内容，只增不改写。**铁律：PRD 不写历史，requirements 不写愿望。** 「考虑过 OIDC 但放弃了」是事实 → requirements；「未来要支持多租户」是主张 → PRD。最常见错：把 PRD 愿望写进 requirements，后续会话会以为「这事说定了」。
- **PRD vs wiki**：PRD 对内、前瞻 intent；wiki 对外、as-built——现在是什么、怎么用。两者在「定位/目标用户/价值」上重叠。**重叠项 PRD 写一次当源，wiki 引用不复制**，否则两处必然漂移过期。
- **PRD 不进 always-on context**：Kiro steering / spec-kit constitution 每次交互全量加载产品文档，docstrata 不学。PRD 与其余层一样走 INDEX 按需加载；grilling 卡在产品决策时才读，不为用不到的内容每次付 token。

PRD 不在 CoALA 三类长期记忆里。它是当下的前瞻主张，四层之上的意图锚点；grilling 答不出某个决策时，用 PRD 的定位/价值/优先级消歧。

## Completeness Contract（信息维度）

生成前确保以下维度达标（探索 + 必要时 grill）：

1. **定位** — 这是个什么产品、解决谁的什么核心问题、与替代方案的根本差异。变更频率最低，几乎不动。
2. **目标用户** — 为谁做，主要用户角色与他们的核心诉求。
3. **价值主张与优先级原则** — 我们优先什么、牺牲什么（如「优先简单而非全面」）。消歧依据：能回答「该选 X 还是 Y」。
4. **功能范围** — 要做的主要功能（意图层，不是 requirements 的验收条目）。每条一句话「做什么 + 为什么值得做」。
5. **非目标** — 明确不做什么、不为谁做。与功能范围同等重要，划清边界。
6. **roadmap** — high-level 推进顺序与当前优先级。变更频率最高，条目带日期。

## 探索重点

从原始材料出发，区分主张与事实，把产品决策与技术决策分流。

- 找原始材料：现有 PRD/产品方案、README 的愿景段、requirements 里被误塞的产品级决策。PRD 缺位时这些常挤在 requirements 的决策记录里，生成 PRD 时上移过来。
- 区分原始主张与倒推：从成品反推的定位标 `[推断]`，让用户确认。
- 产品决策与技术决策分流：「要不要做某功能、定位、对外策略」属 PRD；「这批功能怎么实现、用什么技术」属 requirements/dev。

## Section 骨架

按变更频率从上到下排版：稳的在顶（定位），易变的在底（roadmap 带日期），过期风险一眼可见。

```markdown
# PRD — {产品名}

> {一句话产品主张：这是个什么产品、给谁、解决什么}

last-verified: YYYY-MM-DD

## 定位
{这是什么产品、解决谁的什么核心问题、与替代方案的根本差异。声明式、可判定，少用含糊词}

## 目标用户
{主要用户角色 + 各自的核心诉求}

## 价值主张与优先级原则
{我们优先什么、牺牲什么。写成能用来做决策的原则，而非口号}

## 功能范围
{主要功能，意图层。每条：做什么 · 为什么值得做。不写验收标准（那是 requirements）}

## 非目标
{明确不做什么、不为谁做。划清边界}

## Roadmap
{high-level 推进顺序与当前优先级，条目带日期。这一节变更最频繁}
```

## 写作要求（借 spec-kit constitution）

主张要可裁决，更新要直接重写。

- **声明式、可判定**：定位与优先级原则写成能用来裁决的主张，避免「应该/尽量/灵活」这类含糊词。
- **roadmap 带日期**：每条 roadmap 标注写入/更新日期，让读者识别哪条可能已过期。
- 重写而非追加：PRD 是主张，认知更新时**直接重写**对应段落（不像 requirements 只增）。定位变了就改定位，旧主张不保留为「历史」（历史属 requirements/git）。

## Grill 示例（仅缺口）

- 定位 low：「我从代码/README 推测这个产品是给{X}用来解决{Y}的——定位对吗？和现有替代方案最根本的差异是什么？」
- 优先级原则 missing：「做取舍时你更看重什么？比如『优先简单上手』还是『优先功能完整』？这条会成为以后我帮你做产品决策时的依据。」
- 非目标 low：「有没有明确不做、或明确不服务的对象？这条和功能范围一样重要，能挡住后面的范围蔓延。」
