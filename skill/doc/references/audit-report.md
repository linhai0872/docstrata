# Audit Report — 信息健康诊断副产物

任何生成动作完成后，向用户输出的一份**信息健康诊断报告**。它不是文档，是判断——告诉用户"你项目里的事实信息哪些对不上了"。

## 定位

- **不是独立命令**，是生成流程的副产物。生成 wiki/dev/index/all 时自然带出。
- **不额外生成文档文件**，直接在对话里汇报给用户（除非用户要求落盘）。
- **产出判断，不产出文档**。生成模式已经在写文档了，audit 的独特价值是"信任评估"。
- 对象是**全部 context**，不止文档：代码、git 历史、现有文档、规范文件（CLAUDE.md/AGENTS.md/lint 配置等）之间的事实是否自洽。

## 数据来源

EXPLORE / MAP 阶段按 source-criticism 处理多源信息时，本就会发现一批问题。audit 不重新跑一遍，只是**汇总**这些发现：

- 准则1（可信度排序）筛出的：仅低可信来源支撑、无代码佐证的主张
- 准则2（矛盾处理）标出的：`[过期]` `[冲突]`
- 准则3（事实vs推断）标出的：`[缺口]`
- 探索中遇到的无主孤儿文档（不属于任何层、且与代码无关联的旧文档）

## 报告结构

按严重度排序输出，每条给出**证据**和**建议动作**：

```markdown
## 信息健康诊断

扫描范围：{代码 / git / docs/ / 规范文件，列出实际查了什么}

### 冲突（需人工裁决）
- **{主题}**：{来源A} 说 X，{来源B} 说 Y。
  证据：{文件:行 或 具体依据}
  建议：以 {高可信来源} 为准 / 需你确认

### 过期（与当前事实不符）
- **{文件}**：描述了 {旧方案}，但代码已是 {现状}。
  证据：{对比依据}
  建议：更新 / 归档 / 删除

### 缺口（重要但无人记录）
- **{主题}**：{哪一层的哪个维度} 缺信息。
  建议来源：{人 / 文件 / 代码区域}

### 无主孤儿文档
- **{文件}**：不属于四层任何一类，且与当前代码无对应。
  建议：归入 {某层} / 归档 / 确认是否废弃
```

## 纪律

- **不夸大**：没发现问题就如实说"未发现信息质量问题"，不为凑内容编造。
- **给证据**：每条问题必须可追溯到具体来源，不空泛断言。
- **不自动修**：audit 只诊断和建议，不擅自改/删用户的文档（删改是 Ask First 级操作）。
- **有限度声明**：若因范围/权限只扫了部分 context（如没读 .doc、没查 git blame），明说哪些没覆盖，不假装查全了。
